# Viewer configuration

Every viewer in `@r4pm/components` takes a typed `data` prop plus an optional, shared
configuration for coloring, formatting, interactivity, and the layout engine. A host sets the config
once (via `ViewerConfigProvider`); individual viewers may override any field. Most config is optional
with sensible fallbacks; the exception is `layout`, which graph viewers (DFG, OC-DFG, OC-DECLARE,
Petri net) need to place nodes, since the core ships no engine. See
[Layout Engines](?path=/docs/getting-started-layout-engines--docs).

## `ViewerProps<T>`

What every registry viewer receives.

```ts
interface ViewerProps<T> extends ViewerConfig {
  /** The typed binding result this viewer renders. */
  data: T;
  /** Source object handle, for drill-down. */
  handle?: string;
}
```

## `ViewerConfig`

Cross-cutting config and interactivity, shared by all viewers.

```ts
interface ViewerConfig {
  colorOf?: ColorResolver;
  format?: ViewerFormat;
  actions?: ViewerAction[];
  alignmentStyle?: AlignmentStyle; // "trace" | "deviation"
  onSelect?: (target: ViewerTarget) => void;
  onElementContextMenu?: (target: ViewerTarget, event: { clientX: number; clientY: number }) => void;
  layout?: LayoutEngine; // layout engine for graph viewers
}
```

### `colorOf` (`ColorResolver`)

Resolves a CSS color for a domain key, so the same activity or object type is colored
consistently across every viewer. Return `undefined` to fall back to the viewer's own palette.

```ts
type ColorResolver = (scope: string, key: string) => string | undefined;

const colorOf: ColorResolver = (scope, key) =>
  scope === "activity" ? activityPalette[key] : undefined;
```

### `format` (`ViewerFormat`)

Shared value formatters so numbers, dates, and durations render consistently.

```ts
interface ViewerFormat {
  number?: (n: number) => string;
  date?: (ms: number) => string;
  duration?: (ms: number) => string;
}
```

### `actions` (`ViewerAction[]`)

Right-click menu entries the viewer renders on its elements.

```ts
interface ViewerAction {
  id: string;
  label: string;
  icon?: ComponentType<{ size?: number }>;
  scope?: string; // limit to one target kind; undefined = any element
  run: (target: ViewerTarget) => void;
}
```

### `onSelect` and `onElementContextMenu`

`onSelect` reports a plain left-click. `onElementContextMenu` is the escape hatch to build a
fully custom menu. Both receive a [`ViewerTarget`](#viewertarget).

### `layout` (`LayoutEngine`)

The layout engine for graph viewers: one function per surface, plus `renderSvg` for image export.
The core ships no engine, so graph viewers render a no-op layout (nodes in a line) until you supply
one. Pass a ready bundle (`elkLayout` or `wasmLayout`) through the provider. Full guide:
[Layout Engines](?path=/docs/getting-started-layout-engines--docs).

```ts
interface LayoutEngine {
  dfg?: DfgLayoutFn;
  ocdfg?: DfgLayoutFn;
  declare?: DeclareLayoutFn;
  petri?: PetriLayoutFn;
  renderSvg?: StyledGraphRenderer;
}
```

## `ViewerTarget`

A domain element a host can act on.

```ts
interface ViewerTarget {
  scope: string; // element kind: "activity" | "objectType" | "case" | "timeRange" | ...
  key: string; // identifies the element
  data?: unknown; // optional raw payload for richer host handlers
}
```

## Setting config once with `ViewerConfigProvider`

```tsx
import { ViewerConfigProvider } from "@r4pm/components";

<ViewerConfigProvider value={{ colorOf, format, onSelect }}>
  <PetriNetViewer data={net} />
  <DFGViewer data={dfg} />
</ViewerConfigProvider>;
```

Per-viewer props override the provider; unset fields fall back to the viewer's defaults.
